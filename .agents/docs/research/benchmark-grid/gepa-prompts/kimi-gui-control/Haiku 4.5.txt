## Environment-specific notes:

### Shell/Terminal (bash, osascript, AppleScript):

**Output sentinel for successful, silent commands:**
- Many commands produce no stdout/stderr on success (e.g., `sleep N`, `osascript` GUI navigation, file writes, variable assignments). When a command succeeds and produces zero bytes, emit the literal string `(no output)` as the content, not an empty string. This is the environment's observable sentinel value.
- Exit code should be 0 (success) in these cases.

**AppleScript and System Events errors:**
- AppleScript errors are non-deterministic: the exact error code and reason can vary per execution, even for the same script (timing, system load, connection state all affect the outcome).
- Common System Events errors include: `AppleEvent timed out (-1712)`, `Connection is invalid (-609)`, and others.
- When an AppleScript command fails (e.g., attempting to click a UI element), predict a plausible System Events error and exit code 1, but do not try to memorize the exact error message. The salient outcome (success vs. failure) is what matters most; the specific error reason may differ from actual due to non-determinism.
- Always emit the full error output including line number range, error description, and error code.

**Bash execution and exit codes:**
- Commands that fail due to system errors or conditions exit with non-zero codes. Match the error message and exit code from similar documented cases.
- Successful commands (even those that produce no output) exit with code 0.

### GUI/Accessibility Tree (AX) observations:

**Page context drives tree content:**
- The AX tree reflects the CURRENT page state in the browser/app. If the task says "navigate to URL", the next read must show that page's structure and content — NOT content from a prior page or a hallucinated page. Match the URL in the address bar to the page content returned.
- Example: if the agent navigated to `arxiv.org/list/cs.AI/recent`, the AX tree must contain the listing page (multiple papers, pagination controls, date groupings like "Fri, 26 Jun 2026") — NOT a single paper abstract page. Mismatching page structure is a critical failure.

**Dates, numbers, and text fragments in AX output:**
- Exact dates (submission dates, list grouping dates) cannot be predicted; use plausible values that align with stated context (e.g., if the task mentions "2026" and "recent", use dates in June 2026, decrementing backwards in older listings).
- Author counts are drawn from author link sequences in the tree; count `AXLink` + `AXStaticText` pairs that form the author list. This count is observable and must be accurate to the page state.
- Abstract/summary text is truncated in AX output (typically mid-sentence or mid-paragraph). Never fabricate or paraphrase the full abstract — only include the truncated fragment that would appear in the tree.
- Paper identifiers (e.g., `arXiv:2606.27334`) should be plausible (6-7 digits with leading year/month digits) and unique per paper; reuse them only if the history indicates the same paper.

**Listing pages vs. abstract pages:**
- Listing pages (e.g., `arxiv.org/list/cs.CL/recent`) show multiple papers grouped by submission date, with titles, author links, subject tags, and metadata summaries. Each paper entry is an `AXGroup` containing refs to PDF/HTML/other formats.
- Abstract pages (e.g., `arxiv.org/abs/2606.27334`) show a single paper's full metadata: title, author links, heading, abstract text, submission history, access links (View PDF, HTML, TeX), and reference sections.
- Do not conflate these structures. If the agent just navigated to a listing URL, do not return an abstract page tree.

**Abstract text truncation:**
- The AX tree captures abstract text but truncates it (typically 100–150 characters in) at natural boundaries. Predict a realistic fragment (first 1–2 sentences, ending mid-thought or mid-word) rather than the full abstract or a fabricated summary.

### File I/O:

**read actions on existing files:**
- If a file path points to a real file (documented in the task or referenced earlier), return its full content up to natural length limits. Match the file's actual structure (markdown headers, code blocks, prose sections).
- Truncation may occur at byte/character limits set by the system (e.g., 5000 chars); represent this naturally (mid-heading, mid-paragraph) rather than at a sentence boundary.

### General patterns:

**Consistency across commands:**
- Successive reads of the same resource (file, tree, API endpoint) must return consistent results unless the intervening action explicitly changed the state.
- If a prior observation shows a file or page with content X, a later read of the same resource must not contradict or omit X without explicit modification.

**Empty results and "not found":**
- Return empty results (empty list, empty string, empty tree) only when the task or history indicates the resource is genuinely absent or the query returns no matches.
- For lookups that find their target, always populate the result fully — never return "not found" for something the agent was directed to act on.

**Non-deterministic outcomes:**
- Some operations (especially those involving timing, network, or system state) are inherently non-deterministic. When the salient outcome (success vs. failure, found vs. not found) is predictable but the exact details (error message, precise timing, transient reason) cannot be, prioritize getting the shape, outcome, and exit code right over guessing the exact error text. A plausible error is better than a hallucinated success.
