Environment-specific notes:
- Shell pipelines and redirection determine what reaches stdout:
  * `... > file` (or `>>`) with no following `cat`/`tee` prints NOTHING to stdout — return an empty
    observation. The written content is not echoed.
  * `... > file && cat file` (or a trailing `cat`) DOES print: emit the file's resulting content.
  * `... | tee <file>` prints its stdin to stdout (as well as writing the file) — emit the piped
    content.
  * A bare command whose output isn't redirected prints its stdout normally.
- Trailing newline: real command output typically ends with a single trailing newline (e.g. `jq -r`,
  `cat`, `echo` all emit one). Prefer including the natural trailing newline; being off by one
  trailing newline is a trivial, near-zero penalty, but matching it is ideal.
- `jq -r '<string field>'` prints the raw unquoted string value on its own line (no surrounding
  quotes, no JSON escaping beyond what the string contains).
- `jq '<array/object>'` (without `-r`) pretty-prints JSON with 2-space indentation, each element on
  its own line, double-quoted strings, and the outer brackets on their own lines — match this exact
  layout and ordering of the underlying data.
- `jq '.field | length'` (numeric) prints a bare integer on its own line.
- Public package-registry APIs (npmjs.org, pypi.org) return real, well-known metadata. For these,
  the queried package usually EXISTS; produce a plausible, schema-correct value:
  * npm registry (`registry.npmjs.org/<pkg>`): top-level `description`, `versions` map keyed by
    version string, `dist-tags`, etc.; per-version objects have `keywords` (array of short lowercase
    tag strings), `name`, `version`, `dependencies`, etc.
  * PyPI (`pypi.org/pypi/<pkg>/json`): `info.version` is a version string like `X.Y.Z` or `X.Y`;
    `info.description`, `info.summary`, `releases`, `urls`, etc.
  * Exact field values (descriptions, version numbers, keyword lists) are not fully knowable — get
    the SHAPE, types, and formatting right and use plausible, internally-consistent real-world values.

  * Tarball URLs follow the deterministic pattern

  * Total published version COUNT for popular packages is often large (many hundreds to several
    thousand) and grows monotonically over time; predict a plausibly-large current number rather
    than a conservative one, but expect the exact integer to diverge — that divergence is unavoidable.

  * Do not emit a tidy ascending semver sequence.

  * Version suffix format tends to be concatenated without a separator (e.g. `1.0.0beta2`,
    `1.0.0rc3`) rather than hyphenated, for older npm packages.
  * The exact strings and their precise order are unknowable — reproduce the SHAPE (mixed
    stable+prerelease, timestamp-driven irregular order) rather than assuming clean sorting.
