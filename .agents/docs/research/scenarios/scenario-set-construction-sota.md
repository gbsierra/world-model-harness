# State of the Art (mid-2026): Constructing Representative Eval/Scenario Sets from AI-Agent Telemetry Traces

> Research compiled 2026-07-02 for the scenario-set-construction feature
> (branch `feature/scenario-set-construction`). Web + paper survey; citations inline.

## Executive summary

By mid-2026 this has consolidated into a fairly standard pipeline shape, but no single paper owns it end-to-end. The canonical stack is: **(1) LLM facet extraction per trace → (2) embed the extracted facets (not the raw trace) → (3) density or partitioning clustering (HDBSCAN or k-means/k-medoids) → (4) LLM cluster naming + hierarchy (the Clio pattern, arXiv 2412.13678) → (5) representative selection (medoids, or coverage-driven submodular/facility-location selection) with explicit rare-slice retention → (6) LLM synthesis of each selected trace into a parameterized scenario + per-scenario checklist/rubric (the WildBench pattern, arXiv 2406.04770) → (7) validation via coverage-vs-corpus metrics and correlation with production scores.** Industry tools (LangSmith Engine, Braintrust Loop/Topics, Arize Phoenix, Databricks) all now do some version of "cluster traces → promote clusters to eval datasets," but the clustering is mostly intent/issue-labeling plus HDBSCAN over embeddings; principled representative-subset selection (facility location, DPPs, IRT-based subset compression) still lives mainly in the academic benchmark-compression literature (tinyBenchmarks, Scales++, Fluid Benchmarking, EssenceBench) and is a genuine differentiation opportunity for the world-model harness. The most important known failure modes: whole-trace embeddings wash out task intent (fix: embed LLM-extracted task summaries), frequency-proportional sampling drops rare-but-critical failures (fix: stratified hybrid sampling with failure/severity strata), and scenarios synthesized without verifying replayability drift from what actually happened (fix: closed-loop verification, as in Trajectory2Task, arXiv 2601.20144).

## 1. Published approaches for building evals from traces/logs

**Intent taxonomy induction from usage logs (the anchor method):**
- **Clio: Privacy-Preserving Insights into Real-World AI Use** (Anthropic, arXiv [2412.13678](https://arxiv.org/abs/2412.13678), Dec 2024; [blog](https://www.anthropic.com/research/clio)). The reference design: per-conversation LLM **facet extraction** (task, topic, language, turn count), embed the facet summaries, k-means base clustering, then LLM-generated cluster names/descriptions and **recursive hierarchical clustering** into a multi-level taxonomy, with minimum-cluster-size thresholds. Open reimplementation: [OpenClio](https://github.com/Phylliida/OpenClio). This is the SOTA pattern for steps (1)–(4). Anthropic's later Economic Index work reuses it.
- **WildBench** (AI2, arXiv [2406.04770](https://arxiv.org/abs/2406.04770), Jun 2024). The reference for trace→benchmark distillation: 1,024 tasks selected from 1M+ real WildChat conversations via task-category quotas + multi-LLM difficulty labeling (drop tasks all models find easy), plus **per-instance 5–10 item checklists** generated by GPT-4-turbo + Claude-3-Opus for judging. Closest published precedent for "representative scenario set from production logs."

**Failure-mode mining / trace debugging (feeds "which traces deserve eval slots"):**
- **TRAIL: Trace Reasoning and Agentic Issue Localization** (Patronus AI, arXiv [2505.08638](https://arxiv.org/abs/2505.08638), May 2025). 148 annotated OpenTelemetry traces (GAIA + SWE-bench), 841 errors, 20+ error taxonomy across reasoning/planning/system-execution; best LLM gets ~11% joint localization accuracy. Useful both as a taxonomy and as a public trace corpus ([HF dataset](https://huggingface.co/datasets/PatronusAI/TRAIL)).
- **Why Do Multi-Agent LLM Systems Fail? (MASFT)** (arXiv [2503.13657](https://arxiv.org/abs/2503.13657), Mar 2025) — 14 failure modes in 3 categories, Cohen's κ 0.88, with an LLM-annotator pipeline.
- **Who&When** (arXiv [2505.00212](https://arxiv.org/abs/2505.00212), Apr 2025) — failure attribution to agent+step; 127 MAS failure logs.
- **AgentErrorTaxonomy / AgentErrorBench / AgentDebug** (arXiv [2509.25370](https://arxiv.org/abs/2509.25370), Sep 2025); **AgentFail** root-cause dataset (arXiv [2509.23735](https://arxiv.org/abs/2509.23735), Sep 2025); **CORRECT** — online cache of distilled **error schemata** reused across trajectories (arXiv [2509.24088](https://arxiv.org/abs/2509.24088), Sep 2025); **FAMAS** spectrum-based attribution (arXiv [2509.13782](https://arxiv.org/abs/2509.13782)); **AgentRx** + 115-trajectory failure benchmark (arXiv [2602.02475](https://arxiv.org/abs/2602.02475), Feb 2026); **TELBench/DRIFT** span-level error localization, 2,790 real trajectories → 1,000-instance benchmark (arXiv [2606.02060](https://arxiv.org/abs/2606.02060), Jun 2026).
- **PAEF: Production Agentic Evaluation Framework** (arXiv [2605.01604](https://arxiv.org/abs/2605.01604), May 2026) — taxonomy of 7 production-only failure modes (drift, tool-failure cascades, etc.); argues standard benchmarks miss 4/7 entirely. Directly motivates trace-derived evals.

**Scenario/task synthesis from trajectories:**
- **Trajectory2Task** (arXiv [2601.20144](https://arxiv.org/abs/2601.20144), Jan 2026) — the inverse-direction result most relevant here: explore tool environments to get valid trajectories, then **convert trajectories into user-facing tasks with controlled intent perturbations** (ambiguous / changing / infeasible intent), yielding *verifiable* closed-loop eval+training tasks.
- **ALMITA** — automated test generation for tool-augmented conversational agents (arXiv [2409.15934](https://arxiv.org/abs/2409.15934), Sep 2024) — intermediate graphs from procedures to bound LLM hallucination and enforce conversation-space coverage.
- **TaskBench** back-instruct over tool graphs (arXiv [2311.18760](https://arxiv.org/abs/2311.18760), Nov 2023); **Graph2Eval** KG-based task generation with reachability/LLM-score/similarity filters (arXiv [2510.00507](https://arxiv.org/abs/2510.00507), Oct 2025); **IntellAgent** (arXiv 2501.11067, Jan 2025) — policy-graph-driven scenario generation + simulated users.
- **Docent** (Transluce, [transluce.org/introducing-docent](https://transluce.org/introducing-docent), 2025) — practical system for **summarize / search / cluster / intervene** over agent transcripts; the "extract observations per batch → cluster observations" pattern is a good alternative to embedding-clustering raw traces.

**Surveys for orientation:** *Survey on Evaluation of LLM-based Agents* (arXiv [2503.16416](https://arxiv.org/abs/2503.16416), Mar 2025) and *Evaluation and Benchmarking of LLM Agents* (arXiv [2507.21504](https://arxiv.org/abs/2507.21504), Jul 2025) — the latter explicitly flags trace-derived, continuously updated evals as an open enterprise need.

## 2. Industry tooling: what platforms actually do

- **LangSmith (LangChain).** Long-standing "add run to dataset" from traces + online evals; the 2026 addition is **[LangSmith Engine](https://www.langchain.com/blog/introducing-langsmith-engine)** (public beta): continuously **clusters production failures into named issues**, diagnoses root causes, drafts fixes/evaluators, and **pulls failing traces into ground-truth datasets** so "every resolved issue improves eval coverage." Also "Align Evals" for calibrating LLM judges against human labels. Algorithmically: embedding clustering + LLM labeling; no published medoid/submodular machinery.
- **Braintrust.** One-click trace→eval-case promotion; **Loop** (agent that writes evals/datasets from logs) and **Topics**, which "analyzes production traces by task, sentiment, and issues, then **promotes high-value trace clusters into evaluation datasets**" ([braintrust.dev](https://www.braintrust.dev/articles/best-ai-observability-tools-2026)). Labels every trace with intent/sentiment/issue rather than pure embedding clusters.
- **Arize Phoenix.** Most explicit about algorithms: **UMAP projection + HDBSCAN clustering over embeddings**, clusters ranked by **drift** vs. a baseline; clusters/spans exportable as datasets ([docs](https://arize.com/docs/phoenix)). Discovery-oriented, not representativeness-optimized.
- **Databricks Agent Evaluation / MLflow 3.** [`generate_evals_df`](https://docs.databricks.com/aws/en/generative-ai/agent-evaluation/synthesize-evaluation-set) synthesizes a "representative evaluation set" — but from **documents**, not traces; the trace path is production-log scoring with built-in judges, then promoting logged traces into eval datasets.
- **OpenAI Evals / Agent Evals.** Datasets + trace graders over agent workflows ([guide](https://developers.openai.com/api/docs/guides/agent-evals)); "pull logged completions/traces into datasets, attach graders (including trace-structure graders)." No automatic representative selection.
- **Confident AI / DeepEval.** [Synthesizer](https://deepeval.com/docs/golden-synthesizer): goldens from documents/contexts/scratch/existing-goldens, with **evolution** (DEITA-style complexity mutations) and, since late 2025, **multi-turn goldens** for agent simulation; plus a conversation simulator. Trace→golden conversion exists but selection is manual/filter-based.
- **Langfuse.** Datasets + trace export; deliberately expects curation logic to be external (annotation queues, "add trace to dataset"). Good substrate, no algorithmic curation.
- **HoneyHive / Gentrace / Maxim / LangWatch / Galileo.** Trace filtering → dataset promotion; Galileo does failure clustering; Maxim pushes "convert production traces into reusable simulations + evolve a golden suite for regression checks" ([getmaxim.ai](https://www.getmaxim.ai/articles/the-evolution-of-ai-quality-from-model-benchmarks-to-agent-level-simulation-in-2026/)).

**Takeaway:** industry converged on *cluster→label→promote*; nobody ships principled coverage-optimal selection, frequency-vs-severity weighting controls, or verified scenario abstraction. That's the white space.

## 3. Technical building blocks

**Trace serialization & embeddings.**
- Dominant practice: **don't embed raw traces.** Extract a compact natural-language facet (task intent, 1-sentence summary, tool-sequence signature, outcome) with a cheap LLM, then embed that (Clio does this; Docent extracts observations first). Raw multi-thousand-token traces exceed embedding context and are dominated by boilerplate (system prompts, tool schemas, retrieved docs).
- Common embedders in 2025–26 tooling: OpenAI `text-embedding-3-large`, Cohere embed-v4, `all-mpnet`/`gte`/`bge` families, Qwen3-Embedding; instruction-tuned embedders let you prompt "represent the user's task" to bias geometry toward intent.
- Useful multi-view trick: separate embeddings for (a) intent summary, (b) tool-call sequence (serialize as `tool1→tool2→…` with args stripped), (c) failure/outcome description; cluster per view or concatenate weighted views. Step-level embeddings mostly appear in error-localization work (TELBench spans) rather than in set construction. Generic trajectory-embedding learning exists (arXiv [2501.09327](https://arxiv.org/html/2501.09327v2)) but isn't yet standard for LLM traces.

**Clustering.**
- **k-means on facet embeddings + LLM naming + recursive hierarchy** (Clio) — robust default at scale; k ≈ n/50–n/200 at the base layer, 2–4 hierarchy levels.
- **HDBSCAN (+UMAP for visualization)** — Phoenix's choice; handles variable-density intents and gives an explicit noise bucket (good: the noise bucket is exactly the long tail; review it manually). Caveat: UMAP-then-cluster distorts distances; cluster in ambient space, project only for viz.
- **k-medoids/PAM (CLARA/FasterPAM for scale)** — used where the representative must be a *real trace* (medoid). For 10⁵–10⁶ traces use CLARA-style sampling or BanditPAM.
- Intent-discovery refinements: **NILC** LLM-assisted clustering with semantic centroids and hard-sample rewriting (arXiv [2511.05913](https://arxiv.org/abs/2511.05913), Nov 2025).

**Representative selection (the underexploited layer).**
- **Medoid-per-cluster** — the industry default; simple, auditable.
- **Facility location maximization** (submodular; lazy/stochastic greedy à la Mirzasoleiman et al.) — maximize Σᵢ max_{s∈S} sim(i,s); (1−1/e) coverage guarantee over the *whole* corpus rather than per-cluster; naturally budget-constrained ("pick 200 scenarios"). See also semantic-compression framing for vector search (arXiv [2507.19715](https://arxiv.org/abs/2507.19715)).
- **k-DPP / DPP sampling** for diversity (arXiv [2006.16947](https://arxiv.org/abs/2006.16947)); used in instruction-data selection (arXiv [2402.02318](https://arxiv.org/abs/2402.02318)). MMR (λ·relevance − (1−λ)·redundancy) is the cheap streaming alternative.
- **SemDeDup** (arXiv [2303.09540](https://arxiv.org/abs/2303.09540)) — pre-dedup step: within k-means clusters remove pairs with cosine > ~0.95; cheap and highly effective on production logs where 30–50% of traffic is near-duplicate.
- **Benchmark-compression / model-aware selection:** tinyBenchmarks-line IRT methods; **Fluid Benchmarking** (arXiv [2509.11106](https://arxiv.org/abs/2509.11106)) — IRT + adaptive item selection, 50× fewer items on MMLU with *higher* validity; **Scales++** item-centric cognitive-demand selection, 0.5% subset → 2.9% MAE (arXiv [2510.26384](https://arxiv.org/abs/2510.26384)); **EssenceBench** GA-based compression, 95% rank preservation at 200× reduction (arXiv [2510.10457](https://arxiv.org/abs/2510.10457)); **WILD** multidimensional-IRT + optimal experimental design, 7% MAE from 16 items (arXiv [2604.01418](https://arxiv.org/abs/2604.01418)); older anchors: SMART filtering (arXiv [2410.20245](https://arxiv.org/abs/2410.20245)), small-bench (arXiv [2305.14947](https://arxiv.org/abs/2305.14947)), ONEBench sample-pool aggregation (arXiv [2412.06745](https://arxiv.org/abs/2412.06745)). These require score matrices across models/checkpoints — feasible here because the world-model harness makes re-running cheap.
- **Coverage metrics:** facility-location objective value; % of corpus within cosine τ of ≥1 selected scenario; cluster-mass coverage; **RACC** representation-aware coverage criteria for LLM testing (arXiv [2602.02280](https://arxiv.org/abs/2602.02280), Feb 2026); "Beyond Benchmark Islands" — weighted scenario-manifold coverage + risk sensitivity (arXiv [2603.14987](https://arxiv.org/abs/2603.14987), Mar 2026).

## 4. Scenario abstraction: trace → reusable eval scenario

Consensus recipe assembled from WildBench, Trajectory2Task, ALMITA, and platform docs:

1. **Initial-state extraction:** from the trace's first environment observations and tool results, reconstruct the minimal world state (files, DB rows, ticket contents) needed for the episode. For the world-model harness this becomes the seed context the world model conditions on — a large advantage since no executable sandbox is needed, only a faithful state description.
2. **Task-description synthesis:** LLM rewrites the user's opening (plus revealed constraints from later turns) into a self-contained task statement; PII scrubbed Clio-style. Keep a pointer to the source trace for audit.
3. **Parameterization:** lift entities (names, dates, IDs, quantities) into slots with type constraints so one scenario yields N instantiations; Trajectory2Task additionally applies **controlled intent perturbations** (ambiguous / mid-conversation intent change / infeasible-per-policy) to turn one trace into a family of harder variants.
4. **User simulator spec:** for multi-turn scenarios, derive a persona + hidden-information script from the trace (τ-bench-style). Caution: **Mind the Sim2Real Gap** (arXiv [2603.11245](https://arxiv.org/abs/2603.11245), Mar 2026) ran the full τ-bench protocol with 451 real humans and found LLM user-simulators are excessively cooperative and inflate success rates above human baselines — calibrate the simulator against held-out real traces.
5. **Rubric/judge generation:** per-instance checklists (WildBench); assertion synthesis from prompt-edit history (**SPADE**, arXiv 2401.03038); human-aligned criteria selection (**EvalGen / "Who Validates the Validators"**, arXiv [2404.12272](https://arxiv.org/abs/2404.12272), UIST 2024 — note its **criteria drift** finding: criteria can't be fixed before seeing outputs, so make rubric refinement iterative). For successful source traces, the original trajectory doubles as a reference trajectory for trajectory-match scoring; for failed source traces, encode the failure taxonomy label (TRAIL/MASFT) as a targeted judge question.
6. **Verification loop:** re-run the synthesized scenario (in the world model), check an agent can in principle complete it and the judge's verdict on the original trajectory matches its known outcome. Trajectory2Task and Graph2Eval both treat executability/verifiability filtering as essential; unverified synthesized tasks are frequently unsolvable or under-specified.

## 5. Validation methodology & public corpora

**How "representative" is measured in practice:**
- **Distributional coverage:** fraction of corpus embeddings within τ of the selected set; per-cluster mass covered; facility-location objective; compare selected-set vs. corpus distributions over facets (intent category, tool mix, turn count, outcome) with χ²/JS divergence.
- **Predictive fidelity:** correlation of eval-set scores with full-traffic scores across models/versions — the benchmark-compression standard (MAE of predicted full-score, Spearman/Kendall rank preservation across ≥10 model variants; Fluid, Scales++, EssenceBench all report exactly this). The harness can cheaply generate the model×item score matrix to run this test.
- **Downstream regression detection:** inject known regressions (prompt truncation, tool removal, older model) and measure whether the scenario set flags them; DeepTest-style **failure coverage** (does the set contain a failing case for each known production failure cluster).
- **Judge validity:** correlate LLM-judge verdicts with human labels on a stratified sample (Align Evals, EvalGen); HUMANS benchmark shows well-curated subsets + regression can *out-predict* full benchmarks for user satisfaction (arXiv [2605.00022](https://arxiv.org/abs/2605.00022)).
- **Cluster quality:** silhouette/DB index are reported but weakly trusted; Clio-style validation used LLM-judged cluster-assignment accuracy and reconstruction of known ground-truth topic distributions from synthetic data — a better template.

**Public corpora usable for verification:**
- **WildChat-1M** (arXiv 2405.01470) and **LMSYS-Chat-1M** (arXiv 2309.11998) — real chat traffic; WildBench's 1,024 curated tasks give a reference "distilled set."
- **TRAIL** ([PatronusAI/TRAIL](https://huggingface.co/datasets/PatronusAI/TRAIL)) — OTel agent traces with error annotations (test failure-mode retention).
- **Who&When**, **AgentErrorBench**, **AgentFail**, **CORRECT-Error** (~2k injected-failure trajectories), **TELBench** (2,790 deep-research trajectories) — annotated failure trajectories.
- **τ-bench** (arXiv 2406.12045) / **τ²-bench** (arXiv 2506.07982) — tool-agent-user tasks whose rollout traces can serve as a "production log" with known task identities: run the pipeline on rollouts, check it recovers the task set. The Sim2Real study (arXiv 2603.11245) adds 451-human real interaction data on τ-bench tasks.
- **WebArena** (arXiv 2307.13854) / AgentBench (arXiv 2308.03688) rollout logs; 794 human-labeled WebArena trajectories in arXiv [2603.12710](https://arxiv.org/abs/2603.12710).

A clean verification design: take τ-bench or WebArena rollouts, strip task labels, run the construction pipeline, and measure (a) task-identity recovery (cluster purity vs. true task IDs), (b) rank correlation between the K-scenario set and full-set agent scores across 5–10 agent configs, (c) retention rate of trajectories exhibiting each TRAIL/MASFT failure category.

## 6. Key trade-offs and known failure modes

1. **Whole-trace embedding washes out intent.** Long traces are dominated by tool schemas/retrieved content; two traces with identical scaffolding but different tasks embed nearly identically. Fix: embed LLM-extracted facets (Clio) or multi-view (intent / tool-sequence / outcome). Conversely, intent-only embedding merges "same task, wildly different execution paths" — keep a behavioral view if execution diversity matters for a world-model harness.
2. **Frequency-weighted vs. uniform cluster sampling.** Proportional sampling makes the eval a traffic mirror (good for aggregate-score fidelity) but allocates ~0 slots to rare-critical scenarios; uniform-per-cluster over-weights the long tail and destroys correlation with production metrics. SOTA practice is **stratified hybrid**: report two numbers — a traffic-weighted score (proportional slots) and a coverage/risk score (uniform + severity-boosted slots).
3. **Rare-but-critical retention.** Density clustering routes rare failures to the noise bucket; SemDeDup can delete a unique catastrophic trace that's cosine-close to a benign one. Fix: run failure-mode mining (TRAIL/MASFT taxonomy tagging) *before* dedup/selection and pin one exemplar per (failure-category × severity) regardless of frequency.
4. **Success-only bias.** Datasets promoted from "interesting/failing" traces skew hard; promoted from thumbs-up traces skew easy. WildBench's multi-model difficulty filter (drop unanimously-easy) is the standard correction.
5. **Judge/rubric drift and circularity.** Rubrics generated from the same traces you evaluate encode the incumbent agent's behavior as "correct" (reference-trajectory overfitting — penalizes valid alternate strategies). EvalGen's criteria-drift finding says rubric authoring must stay human-in-the-loop and iterative. Keep rubrics outcome-based where possible; use trajectory-match only as a diagnostic.
6. **Simulator fidelity.** Scenario replay via LLM user simulators inflates success (Sim2Real gap, arXiv 2603.11245) — for this harness the risk applies doubly (world model + user model both smooth over real-world messiness). Calibrate world-model responses against held-out real tool outputs from the trace corpus.
7. **Staleness/drift.** Trace distributions shift; a one-shot eval set decays. Phoenix ranks clusters by drift; PAEF argues for continuous evaluation on production traffic with periodic re-selection. Plan for scheduled re-clustering with cluster-ID stability (match new↔old clusters by centroid similarity) so metrics remain comparable.
8. **k-means vs. k-medoids vs. HDBSCAN.** k-means centroids aren't real traces (need nearest-real-point post hoc); k-medoids is O(k(n−k)²) for PAM — use FasterPAM/CLARA/BanditPAM above ~50k traces; HDBSCAN's noise bucket needs an explicit handling policy or you silently drop 10–30% of traffic.

## Recommended design space

**Design A — "Clio→Medoid→WildBench" (pragmatic default).**
Facet extraction (cheap LLM: task summary ≤30 words, tool-sequence signature, outcome, failure tag via TRAIL/MASFT taxonomy) → embed task summary → SemDeDup at cosine 0.95 → k-means (k≈√n or n/100) + LLM cluster naming + one hierarchy level → per-cluster medoid (nearest real trace to centroid) with **hybrid allocation** (70% of budget proportional to cluster mass, 30% uniform + pinned failure exemplars) → LLM scenario synthesis (initial state, parameterized task, per-instance checklist) → world-model replay verification.
*Pros:* every component battle-tested; auditable (each scenario traces to a real trace); ships fast. *Cons:* k is arbitrary; coverage is per-cluster not global; no model-aware compression; medoids can be mediocre exemplars of heterogeneous clusters.

**Design B — "Global coverage via submodular selection" (better representativeness guarantees).**
Same facet/embedding/dedup front-end, but skip hard clustering for selection: lazy-greedy **facility location** over the full similarity graph with budget K, kernel = weighted sum of intent-sim and tool-sequence-sim; add an MMR/DPP diversity term; enforce matroid constraints (≥1 item per failure-category, per top-level intent, per outcome class). Clustering still runs, but only for naming/reporting.
*Pros:* (1−1/e) whole-corpus coverage guarantee; budget is the natural knob; constraints make rare-critical retention explicit rather than a sampling hack; degrades gracefully as traffic drifts. *Cons:* O(nK) similarity evaluations (fine to ~1M with FAISS + stochastic greedy); less intuitive to explain than "one per cluster"; kernel weighting needs tuning.

**Design C — "Model-aware compressed eval" (highest fidelity, uses the harness's superpower).**
Start from Design A or B's candidate pool (2–5× final budget), then exploit cheap world-model rollouts: run 5–15 agent variants (models/prompts/tool configs) over all candidates, fit a (multidimensional) IRT model, and select the final set by Fluid-Benchmarking-style adaptive item selection / optimal experimental design (WILD) to maximize predicted-full-score fidelity and rank preservation; validate by MAE + Kendall τ against the full candidate pool.
*Pros:* directly optimizes the thing you'll be judged on (does the small set predict production behavior); published results show 25–200× compression with <5% error; only feasible *because* the world model makes the score matrix cheap. *Cons:* model-centric selection can overfit to failure patterns of today's agents (Scales++'s critique — mitigate by blending item-centric diversity from B); more moving parts; IRT needs binarizable or graded scores from judges.

**Recommendation:** ship A as the MVP, evolve the selection layer to B (drop-in replacement for the medoid step), add C as an optional "compress & calibrate" pass once the harness produces score matrices. Validate throughout with the τ-bench-rollout recovery test and TRAIL failure-retention rate from §5.
